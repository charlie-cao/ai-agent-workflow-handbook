## 🟢 Intermediate Tier

### S-01: URL 短链接服务
**类别**: 读写密集型 | **预计对话轮次**: 5-7轮

**背景描述**:  
设计一个 URL 缩短服务（类似 bit.ly）。用户粘贴一个长 URL，系统生成一个短链接（如 `sho.rt/xk2p9`），任何人访问短链接时被重定向到原始 URL。

**规模约束**:
- 每天创建 1 亿条新短链接
- 每天 100 亿次重定向请求
- 读写比 = 100:1
- 短链接永久有效（除非用户手动删除）

**隐藏挑战（仅 AI 知道）**:
1. Hash 碰撞处理策略（MD5 截断 vs 分布式 ID 生成器）
2. 热门 URL 缓存的 Hot Key 问题（某条链接被病毒式传播）
3. 自定义短链接 + 冲突处理
4. 点击数统计的实时 vs 最终一致性设计
5. 数据库选型（读多写少场景下 MySQL + Redis vs Cassandra）

2. 图片缩略图生成（同步 vs 异步，多尺寸）
3. 信息流生成策略（Pull vs Push vs 混合）
4. 热账号（明星用户）粉丝数千万的 Fanout 问题
5. 内容审核管道设计

### S-03: 分布式速率限制器
**类别**: 基础设施 | **预计对话轮次**: 5-6轮

**背景描述**:  
为 API 网关设计一个速率限制系统，防止任何单个客户端滥用 API（如 DDoS、爬虫）。支持多种限制策略（每分钟请求数、每天请求数）。

**规模约束**:
- 服务 1000 个 API 端点
- 每秒处理 10 万次 API 请求
- 限制规则实时生效（变更后 < 1 秒生效）
- 允许轻微的速率超限（< 0.1%）

**隐藏挑战（仅 AI 知道）**:
1. 四种算法对比：固定窗口 / 滑动窗口 / 令牌桶 / 漏桶
2. 分布式环境下的 race condition（多节点同时计数）
3. Redis 原子操作 (INCR + EXPIRE) vs Lua 脚本
4. 用户标识策略（IP / API Key / User ID，各有缺陷）
5. 处理 Redis 故障时的降级策略（熔断？放开限制？）

### S-04: 键值存储数据库
**类别**: 存储引擎 | **预计对话轮次**: 6-8轮

**背景描述**:  
设计一个类似 Redis 的键值存储数据库（内存中），支持 GET/SET/DELETE 操作，支持 TTL（过期时间），需要支持数据持久化。

**规模约束**:
- 读写延迟 < 1ms (P99)
- 支持 10 亿个键
- 99.99% 可用性
- 在服务器重启后数据不丢失

**隐藏挑战（仅 AI 知道）**:
1. 内存数据结构选型（Hash Table vs SkipList for sorted sets）
2. 持久化策略：AOF（Append-Only File）vs RDB（快照）权衡
3. LRU/LFU 内存淘汰策略实现
4. 并发安全：单线程事件循环 vs 多线程 + 锁
5. 集群分片策略（一致性哈希 vs 哈希槽）

### S-05: 任务调度系统
**类别**: 定时任务 + 队列 | **预计对话轮次**: 5-7轮

**背景描述**:  
设计一个分布式任务调度系统，支持一次性任务（在指定时间执行）和周期性任务（Cron）。类似 AWS EventBridge 或 Celery Beat。

**规模约束**:
- 每天执行 1 亿个任务
- 调度精度 ±1 秒
- 任务执行支持重试（最多3次）
- 99.9% 可用性，任务不能丢失

**隐藏挑战（仅 AI 知道）**:
1. 时间轮算法（Timing Wheel）vs 优先队列
2. 分布式锁防止任务重复执行（Redlock 正确使用）
3. Worker 崩溃后任务恢复（At-least-once vs Exactly-once 语义）
4. 任务依赖关系（DAG 调度）
5. 时区处理的正确性

### S-12: 分布式唯一 ID 生成器

4. 生成服务本身的高可用设计

### S-15: 分布式文件存储系统（类 HDFS/GFS）
**类别**: 存储系统 | **预计对话轮次**: 8-10轮

**背景描述**:

**隐藏挑战（仅 AI 知道）**:
1. 主节点元数据热点（NameNode 单点问题及其联邦方案）
2. 大文件分块（Chunk）策略与跨节点检索效率
3. 副本放置策略（机架感知 Rack-Aware Replication）
4. 顺序写优化 vs 随机访问权衡
5. 崩溃恢复：重放日志（Edit Log）vs 检查点（FsImage）

**规模约束**:
- 1 万个企业租户，每个租户最多 10 万用户
- 每次 API 请求需要权限校验（< 5ms 加入总响应时间）
- 支持策略变更 < 2 秒生效（跨所有服务）
- 权限决策需完整审计日志

**隐藏挑战（仅 AI 知道）**:
1. 权限策略存储：LDAP 树形结构 vs 策略引擎（OPA/Casbin）
2. 权限检查的缓存策略（策略变更后缓存失效的范围）
3. 租户数据隔离：行级安全（Row-Level Security）vs Schema 隔离 vs 独立 DB
4. OAuth 2.0 + OIDC 集成与租户自定义 IdP（企业 SSO）
5. 细粒度资源权限（某用户只能查看某些行的数据）的性能设计
