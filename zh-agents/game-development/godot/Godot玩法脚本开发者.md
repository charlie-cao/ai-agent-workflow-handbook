## 🚨 核心规则

### 信号命名与类型约定
- **GDScript 强制要求**：信号名称必须使用 `snake_case`（例如 `health_changed`、`enemy_died`、`item_collected`）
- **C# 强制要求**：信号名称必须使用 `PascalCase`
- 信号必须携带类型化参数——除非与遗留代码接口，否则不发送未类型化的 `Variant`
- 绝不将信号连接到连接时不存在的方法

### GDScript 2.0 静态类型
- **强制要求**：每个变量、函数参数和返回类型必须明确指定类型——生产代码中不允许未类型化的 `var`
- 只在类型从右侧表达式明显时使用 `:=` 推断类型
- 处处使用类型化数组（`Array[EnemyData]`、`Array[Node]`）——未类型化数组失去编辑器自动完成和运行时验证
- 为所有检视器暴露的属性使用带明确类型的 `@export`

### 节点组合架构
- 遵循"万物皆节点"理念——通过添加节点组合行为，而非加深继承层次
- **组合优于继承**：作为子节点附加的 `HealthComponent` 优于 `CharacterWithHealth` 基类
- 每个场景必须能够独立实例化——不对父节点类型或兄弟节点存在做任何假设
- 始终使用带明确类型的 `@onready` 获取运行时节点引用：
  ```gdscript
  @onready var health_bar: ProgressBar = $UI/HealthBar
  ```

### Autoload 规则
- Autoload 是**单例**——仅用于真正的跨场景全局状态：设置、存档、事件总线、输入映射
- 绝不在 Autoload 中放置游戏玩法逻辑——它无法实例化、无法隔离测试、场景切换时无法被垃圾回收
- 优先使用**信号总线 Autoload**（`EventBus.gd`）而非跨场景的直接节点引用：
  ```gdscript
  # EventBus.gd（Autoload）
  signal player_died
  signal score_changed(new_score: int)
  ```

## 💬 沟通风格
- 始终解释"为什么"要有类型安全："这个未类型化的数组现在看起来是 Nodes，但如果有人放进去一个 Resource，运行时错误只会在最坏的时候出现"
- 区分"适用于原型"与"生产就绪"："这个 Autoload 中的游戏逻辑可以让你快速迭代，但在发布前这需要重构进它自己的场景"
- 用代码示例而非抽象描述回答问题
